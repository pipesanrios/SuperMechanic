ACTUALIZACIÓN DE DOCUMENTACIÓN Y CIERRE DE FASE

Acabamos de terminar una fase, subfase, fix o bloque de trabajo del plugin Super Mechanic.

==================================================
CONTEXTO DE CIERRE
==================

La implementación ya fue realizada, corregida y validada previamente.

Objetivo de este prompt:

- consolidar el cierre documental final
- corregir desalineaciones entre código, docs y contextos AI
- dejar una narrativa técnica coherente para la siguiente ventana o sesión

Estado esperado:
El trabajo debe quedar marcado como COMPLETA solo si no quedan inconsistencias relevantes entre código y documentación afectada.

Reglas de alcance:

- no rehacer implementación
- no ampliar alcance
- no reauditar todo el plugin
- no convertir el cierre documental en una nueva fase de desarrollo
- usar el código real actual como fuente principal de verdad
- partir de los archivos realmente modificados por el trabajo ya realizado

No releer todo el proyecto desde cero salvo que detectes una desalineación importante que lo haga necesario.

==================================================
VALIDACIÓN PREVIA OBLIGATORIA
=============================

Antes de actualizar documentación debes:

1. revisar el código realmente implementado en este trabajo
2. comparar los cambios con la documentación actual
3. identificar archivos, clases, tablas y módulos realmente afectados
4. verificar si hubo cambios reales en narrativa operativa, fases, módulos activos, riesgos o deuda técnica
5. usar el código real del proyecto como fuente principal de verdad

Debes confirmar y reportar explícitamente:

- qué archivos fueron realmente modificados en el trabajo cerrado
- qué archivos nuevos se crearon, si aplica
- que el código compila o al menos pasa validación sintáctica disponible
- que no hay errores de sintaxis PHP en los archivos relevantes
- que el bootstrap del plugin no se rompió
- qué validaciones sí se ejecutaron
- qué validaciones no se pudieron ejecutar

Aclaración importante:

“bootstrap no roto” significa como mínimo:

- validación sintáctica del entry point principal del plugin
- validación sintáctica de los archivos PHP modificados

Si no puedes validar la carga completa del plugin dentro de WordPress, debes decirlo explícitamente.
No inventar validaciones no ejecutadas.

Solo después actualizar la documentación.

==================================================
TAREAS DE CIERRE
================

1. ARCHITECTURE.md

- agregar o completar la entrada correspondiente en la bitácora de fases o trabajo
- registrar archivos creados o modificados
- registrar clases nuevas o ampliadas
- registrar tablas nuevas o modificadas
- registrar integraciones realizadas
- actualizar estado del proyecto si cambió realmente

2. FINAL_ARCHITECTURE_MAP.md

- actualizar módulos afectados

- reflejar el estado técnico real actual

3. SYSTEM_MAP.md

- actualizar módulos activos o afectados
- actualizar clases clave
- actualizar tablas por módulo si aplica
- actualizar menús, shortcodes o entry points si aplica
- actualizar riesgos arquitectónicos si aparecieron o cambiaron

4. CURRENT_STATE.md

Actualizar si aplica:

- fase o bloque completado
- estado general del sistema
- módulos activos
- módulos parciales
- últimos cambios técnicos
- riesgos actuales
- pendientes inmediatos

5. MODULE_REGISTRY.md

- registrar módulos nuevos o modificados
- actualizar clases principales
- actualizar dependencias
- actualizar estado del módulo

6. DATABASE_MAP.md

- registrar tablas nuevas o modificadas
- actualizar relaciones si aplica
- actualizar reglas de integridad si aplica
- corregir desalineaciones con el código real

7. DEV_GUIDE.md

- actualizar solo si cambiaron reglas, convenciones o criterios de implementación

8. TEST_SCENARIOS.md

- actualizar solo si cambió el comportamiento funcional real, la cobertura de escenarios o el estado de un escenario

9. TASK FILE RELACIONADO

- actualizar estado: completado / parcial / bloqueado
- registrar notas técnicas finales
- registrar desviaciones respecto al alcance original si existieron
- registrar deuda técnica abierta
- registrar lo que quedó deliberadamente fuera de alcance

10. README.TXT

- actualizar solo si cambió versión, changelog o alcance funcional visible

11. AGENTS.md

- actualizar solo si cambiaron reglas operativas de trabajo con IA

12. CONTEXTOS AI Y PROMPTS

- actualizar solo si el trabajo cambió de forma real:
  - fases o subfases reconocidas
  - módulos activos
  - deuda técnica viva
  - narrativa operativa base
  - criterios de cierre o consistencia
- mantener consistencia entre docs técnicos, contextos rápidos y prompts
- evitar duplicar narrativa desactualizada en varios archivos

==================================================
REGLAS DE DOCUMENTACIÓN
