CIERRE DE FASE — CONTRACT DRIVEN (SUPER MECHANIC)

Se ha completado una fase/subfase/bloque dentro de un Task Contract.
Este prompt es para cierre normal de fase, no para auditoría documental amplia.

==================================================
PRECONDICIÓN OBLIGATORIA
==================================================

Debe existir Task Contract para la tarea.
Sin contract no se permite cierre normal.

El contract debe aportar explícitamente:
- `scope`
- `acceptance_criteria`
- `validations_required`
- `docs_to_update`
- `technical_debt`
- `validation_contract` (si existe para la fase)

Si falta alguno:
→ bloquear cierre y reportar pendiente.

==================================================
FUENTE DE VERDAD (ORDEN OBLIGATORIO)
==================================================

1. código real
2. task contract (define alcance y criterios)
3. `docs/CURRENT_STATE.md` + `.vscode/AI_CONTEXT.md`
4. documentación relacionada al alcance del contract

Si hay conflicto:
→ manda el código real y se ajusta dentro del alcance contractual.

==================================================
REGLAS CONTRACT-DRIVEN
==================================================

- no redefinir alcance
- no ampliar alcance funcional
- no auditar todo el repositorio salvo que el contract lo exija
- actualizar solo documentos incluidos en `docs_to_update`
- registrar deuda técnica según `technical_debt` del contract

==================================================
VALIDACIÓN OBLIGATORIA
==================================================

Debes verificar y reportar:

- archivos realmente modificados
- archivos nuevos
- validaciones obligatorias ejecutadas (incluyendo QA Runner si aplica)
- validaciones obligatorias no ejecutadas (con motivo)
- estado de validación runtime real (si aplica)
- cumplimiento de `acceptance_criteria` del contract
- verificación de que los cambios realizados están dentro del `scope` del contract
- ejecución de `validation_contract` (si fue definido)
- resultado final de validacion contractual (pass/fail)

Reglas:

- Si el contract exige runtime y no se ejecutó:
  → estado final no puede ser COMPLETA

- Si no se cumple `acceptance_criteria`:
  → estado final no puede ser COMPLETA

- Si existe `validation_contract` y falla:
  → estado final no puede ser COMPLETA

- Si QA Runner reporta FAIL en checks obligatorios:
  → estado final no puede ser COMPLETA

- Si hay cambios fuera del `scope`:
  → reportar desviación
  → registrar como deuda técnica o error de ejecución

==================================================
ACTUALIZACIÓN DOCUMENTAL (ACOTADA)
==================================================

Actualizar solo:

- documentos incluidos en `docs_to_update` del contract
- `docs/tasks/<task>.md` si el contract lo exige

No ejecutar auditoría documental global desde este prompt.

==================================================
FORMATO DE RESPUESTA (MÍNIMO)
==================================================

0. contract usado
0.1 validation contract usado (si aplica)
1. archivos modificados
2. archivos nuevos
3. validaciones realizadas
4. validaciones no ejecutadas
5. docs actualizadas
6. deuda técnica abierta
7. resultados validation contract (pass/fail)
8. resultado QA Runner (si aplica)
9. estado final: COMPLETA / PARCIAL / BLOQUEADO

Nota crítica:
- PASS técnico NO implica cierre automático de fase.
- La validación runtime manual sigue siendo obligatoria cuando el contract lo exija.
- Si aplica, ejecutar: `php scripts/qa-runner.php --contract=<ruta> --output=text`.
