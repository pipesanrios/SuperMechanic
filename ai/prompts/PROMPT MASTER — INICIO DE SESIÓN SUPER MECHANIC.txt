==================================================
REGLAS CRÍTICAS DE ARQUITECTURA
==================================================

- La arquitectura activa real está en `includes/*`
- `includes/modules/*` es legacy y no debe extenderse
- SQL solo en repositories
- NO cambiar schema salvo instrucción explícita
- NO duplicar lógica existente
- Reutilizar services siempre que existan
- Mantener separación clara: Controller → Service → Repository

==================================================
REGLAS CRÍTICAS DE SEGURIDAD
==================================================

- NO exponer `file_url` directo
- Usar siempre `Document_Service` + `Download_Service`
- Respetar ownership y visibilidad (Access_Control_Service)
- Validar accesos antes de mostrar o descargar datos
- Mantener nonces en formularios
- No abrir endpoints sin control de acceso

==================================================
REGLAS CRÍTICAS DE UI / UX
==================================================

- NO romper formularios existentes
- NO eliminar nonces ni query args
- NO romper rutas de descarga segura
- Mantener compatibilidad con acciones actuales

- Reutilizar sistema visual existente:
  - assets/css/*
  - clases `sm-*` (cards, badges, buttons, tables, layout)

- NO introducir frameworks externos (React, Vue, etc.)
- NO crear múltiples sistemas visuales paralelos

==================================================
REGLAS DE ESCALABILIDAD
==================================================

- Evitar services "god class"
- Mantener lógica pesada en repositories (SQL)
- Services deben orquestar, no concentrar todo
- Controllers solo presentan/reciben input

- Cualquier lógica reusable debe:
  - vivir en un service existente o helper claro
  - no duplicarse en múltiples módulos

- Preparar el código para:
  - múltiples tipos de negocio
  - configuraciones por taller (settings)
  - crecimiento modular
