# TASK CONTRACT - 57D Async Queue Foundation

## Objective
Create a passive async queue foundation for future SaaS jobs without activating real workers, persistence, cron, external queue providers or background execution.

## Scope
- passive queue job contract
- passive queue context
- passive dispatcher/result models
- future async job vocabulary
- retry/status model documentation
- SaaS architecture and roadmap documentation updates

## In Scope
- queue job normalization structure
- supported future job types
- passive payload validation
- normalized dispatch result
- conceptual retry/backoff model
- documentation for deferred workers/persistence/runtime activation

## Out of Scope
- real queue workers
- cron registration
- database schema changes
- job persistence
- external queue providers
- background execution
- CRM/process/payment/users/API/frontend runtime changes

## Allowed Files
- includes/saas/*
- includes/services/* only if strictly needed for passive queue contracts
- docs/*

## Forbidden Files
- CRM
- process/payment logic
- users/roles
- API controllers
- frontend/assets
- database/schema files
- real cron activation
- external queue services

## Constraints
- passive foundation only
- no `$wpdb`
- no SQL
- no schema/database changes
- no cron/hook registration
- no external HTTP calls
- no job execution
- preserve current runtime behavior
- preserve existing active queue module behavior

## Deliverables
- `includes/saas/class-queue-job-contract.php`
- `includes/saas/class-queue-context.php`
- `includes/saas/class-queue-dispatcher.php`
- `includes/saas/class-queue-result.php`
- `docs/ASYNC_QUEUE_FOUNDATION.md`
- `docs/tasks/2026-04-57d-async-queue-foundation.md`
- current state, roadmap, QA and SaaS architecture docs aligned

## Acceptance Criteria
- queue job structure includes:
  - `job_id`
  - `job_type`
  - `business_id`
  - `tenant_id`
  - `payload`
  - `status`
  - `attempts`
  - `max_attempts`
  - `scheduled_at`
  - `created_at`
  - `last_error`
- supported future job types include:
  - `inventory_import`
  - `inventory_connector_sync`
  - `email_delivery`
  - `webhook_delivery`
  - `google_calendar_sync`
  - `pdf_generation`
- supported statuses include:
  - `pending`
  - `running`
  - `completed`
  - `failed`
  - `retry_scheduled`
  - `cancelled`
- dispatcher validates and normalizes jobs but does not persist or execute them.
- retry/backoff model is defined conceptually only.
- no cron, workers, schema changes, external HTTP calls, API changes or runtime hooks are introduced.
