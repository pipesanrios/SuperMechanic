PHASE: 42B
NAME: Reasignación Operativa Controlada

==================================================
OBJETIVO
==================================================

Permitir reasignar carga operativa de forma manual y controlada
desde dashboard, usando propuestas ya generadas por 41D.

==================================================
SCOPE
==================================================

- ejecutar reasignación real manual
- limitar alcance a items operativos compatibles
- usar confirmación explícita
- registrar resultado visible en dashboard

==================================================
OUT OF SCOPE
==================================================

- autoasignación
- cron
- nuevas tablas
- cambios masivos
- emails / WhatsApp / push
- tocar CRM Pipeline core
- mutaciones fuera de entidades soportadas

==================================================
ALLOWED FILES
==================================================

- includes/dashboard/class-workload-service.php
- includes/dashboard/class-admin-dashboard-controller.php
- includes/crm/class-crm-task-service.php
- includes/crm/class-crm-task-repository.php
- docs/contracts/42B.md
- docs/contracts/validation/42B-validation.md

==================================================
FORBIDDEN FILES
==================================================

- includes/modules/*
- schema/database
- includes/processes/*
- includes/appointments/*
- invoices/payments core

==================================================
OBJETIVO FUNCIONAL
==================================================

Permitir ejecutar reasignación manual segura sobre carga operativa.

Primera versión soportada:
- reasignación de CRM tasks asignadas a usuario

No soportar aún:
- procesos
- citas
- entidades sin owner/asignación clara

==================================================
CRITERIO DE IMPLEMENTACIÓN
==================================================

- reutilizar 41D
- no inventar nueva capa de asignación
- solo ejecutar cuando exista propuesta válida
- validar permisos
- validar usuarios origen/destino
- validar ownership / business_id
- mutación controlada solo sobre entidad soportada

==================================================
FLUJO
==================================================

1. dashboard muestra propuesta
2. usuario ejecuta acción manual
3. sistema valida:
   - proposal válida
   - item soportado
   - destino válido
4. sistema reasigna task
5. dashboard refleja nuevo estado

==================================================
ACCEPTANCE CRITERIA
==================================================

- existe ejecución manual de reasignación
- solo sobre tasks soportadas
- no hay mutación no controlada
- validaciones de seguridad completas
- no regresión sistema

==================================================
VALIDATIONS REQUIRED
==================================================

- php lint
- QA runner
- runtime manual
- validación de reasignación real
