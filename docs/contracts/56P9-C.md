# TASK CONTRACT — 56P9-C Calendar architecture consolidation

## Objective
Audit and consolidate Google Calendar integration services to define a clean separation between internal sync logic and external API integration.

## Scope
- service audit
- class responsibility separation
- canonical service definition
- safe renaming/restructuring

## In Scope
- audit existing services:
  - Services\Google_Calendar_Sync_Service
  - Integrations\Google_Calendar\*
- define canonical sync service
- convert integration layer into API client service
- ensure no duplication of responsibilities

## Out of Scope
- OAuth execution
- real API calls
- UI changes
- cron/queue
- schema changes

## Allowed Files

- includes/integrations/google-calendar/*
- includes/helpers/* (if strictly needed)
- docs/CURRENT_STATE.md
- docs/QA_REPORT.md

## Forbidden Files

- CRM
- reset engine
- API layer
- dashboard controllers

## Constraints
- no wrapper temporary solution
- no duplicated logic
- clear separation:
  - Service = business logic
  - Integration = external API client

## Deliverables
- canonical sync service confirmed
- integration service renamed/reoriented
- no duplicated responsibilities
- docs updated

## Acceptance Criteria
- `Services\Google_Calendar_Sync_Service` remains the canonical internal sync/payload service.
- calendar payload building and validation live only in `includes/services/class-google-calendar-sync-service.php`.
- the integration Google Calendar layer is reoriented as external-client-only architecture for future OAuth/API calls.
- no OAuth execution, token exchange, real Google API calls, UI changes, cron/queue changes or schema changes are introduced.
- no duplicated calendar payload-building responsibility remains between `includes/services/*` and `includes/integrations/google-calendar/*`.
