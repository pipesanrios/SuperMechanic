# TASK CONTRACT - 57G-A Manual Queue Worker

## Objective
Implement a controlled manual queue worker for persisted SaaS jobs without cron, scheduled execution, external calls or real provider execution.

## Scope
- Manual worker class for SaaS queue jobs.
- Repository claim/status helpers needed by the worker.
- One-job-at-a-time processing.
- Simulation-only processing for `inventory_connector_sync`.
- Runtime smoke validation through CLI-like/manual invocation.
- Documentation updates.

## In Scope
- `includes/saas/class-queue-worker.php`
- `Queue_Job_Repository` claim/status helper methods.
- Manual worker result payloads.
- Runtime smoke for valid simulation job and invalid job.
- Current state, QA, roadmap, async queue and SaaS foundation docs.

## Out of Scope
- cron activation
- scheduled workers
- batch processing
- real connector execution
- real provider APIs
- catalog imports
- email sending
- PDF generation
- Google Calendar calls
- REST/admin UI endpoint
- frontend/assets
- CRM/users/process/payment logic

## Allowed Files
- includes/saas/*
- docs/*

## Forbidden Files
- CRM
- users/roles
- process/payment logic
- API controllers
- frontend/assets
- external integrations
- real provider APIs
- cron activation
- scheduled workers

## Constraints
- SQL is allowed only in `Queue_Job_Repository`.
- Worker must not register hooks, cron or schedules.
- Worker must not execute automatically.
- Worker processes at most one job per `process_next()` call.
- Lock token is required for claiming/running.
- 57G-A supports only simulation-safe `inventory_connector_sync`.
- Unsupported or unsafe jobs must fail safely.
- No real catalog writes, external calls, email sending, PDF generation or Google Calendar calls.

## Deliverables
- `includes/saas/class-queue-worker.php`
- Repository helper methods when needed:
  - `get_next_available_job()`
  - `claim_job($job_id, $lock_token)`
  - `release_lock($job_id)`
  - `update_attempts($job_id, ...)`
  - `mark_running($job_id, $lock_token)`
- Worker methods:
  - `process_next()`
  - `process_job(array $job)`
  - `handle_inventory_connector_sync(array $job)`
  - `fail_job($job_id, $error)`
  - `complete_job($job_id, $result)`
- Runtime smoke evidence.
- Documentation updated.

## Acceptance Criteria
- Worker claims one available job with a lock token.
- Valid simulation `inventory_connector_sync` transitions `pending -> running -> completed`.
- Invalid/unsafe job transitions to `failed`.
- Lock token is stored while running and released after completion/failure.
- No cron, scheduled event, hook, external API, real provider execution, catalog import, email sending, PDF generation, API endpoint or frontend asset is introduced.
