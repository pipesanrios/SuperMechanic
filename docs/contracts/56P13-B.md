# TASK CONTRACT — 56P13-B Generic connector contract

## Objective
Define the generic technical contract that all future external inventory connectors must follow in Mekvort.

## Scope
- connector contract documentation
- normalized payload definition
- adapter responsibilities
- validation and error model
- sync operation contract

## In Scope
- connector interface/contract definition
- normalized inventory payload
- required/optional fields
- dry-run requirements
- import/sync requirements
- error handling model
- logging expectations
- business scope requirements

## Out of Scope
- real connector implementation
- external API calls
- OAuth
- scheduled sync
- admin UI
- schema changes
- runtime code changes

## Allowed Files
- docs/*
- ai/*
- architecture docs
- roadmap/state docs

## Forbidden Files
- includes/*
- assets/*
- database/schema changes
- API implementation
- CRM/users/process logic

## Constraints
- documentation-only phase
- no runtime implementation
- must align with `docs/INVENTORY_CONNECTOR_ARCHITECTURE.md`
- must preserve existing catalog/import logic

## Validation Contract
- `docs/contracts/validation/56P13-B-validation.md`

## Acceptance Criteria
- Generic connector identity fields are documented:
  - `connector_key`
  - `provider_name`
  - `provider_type`
  - `version`
  - `business_id`
- Adapter responsibilities and expected methods are documented:
  - `get_connector_key()`
  - `validate_credentials()`
  - `fetch_inventory()`
  - `normalize_item()`
  - `dry_run()`
  - `sync()`
- Normalized inventory payload is documented with required and optional fields.
- Validation rules are documented for required fields, year, mileage, price, currency, stock status and business scope.
- Sync operations are documented:
  - `dry_run`
  - `import_new`
  - `update_existing`
  - `deactivate_stale`
  - `skip_invalid`
  - `conflict_detected`
- Standard error model is documented.
- Logging expectations are documented.
- Conflict handling and retry-safe behavior are documented.
- Security requirements for per-business credentials and no credential leakage are documented.
- Future implementation phases 56P13-C through 56P13-E are aligned.
- No runtime code, schema, `includes/*` or `assets/*` changes are made.

## Deliverables

- roadmap/state updates
