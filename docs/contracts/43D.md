PHASE: 43D
NAME: Sistema de seguridad y rollback

==================================================
OBJETIVO
==================================================

Agregar una capa de seguridad operativa y rollback controlado
para acciones ejecutadas desde la automatización manual,
semi-automática y automática controlada.

==================================================
SCOPE
==================================================

- guardrails previos a ejecución
- rollback controlado para acciones soportadas
- trazabilidad visible de ejecución y reversión
- validación de riesgo antes de mutar datos

==================================================
OUT OF SCOPE
==================================================

- rollback universal de todo el sistema
- nuevas tablas
- cron
- persistencia avanzada de auditoría
- rollback de procesos/citas
- tocar CRM Pipeline core
- auto-recuperación avanzada

==================================================
ALLOWED FILES
==================================================

- includes/automation/class-operational-rules-service.php
- includes/dashboard/class-workload-service.php
- includes/dashboard/class-admin-dashboard-controller.php
- includes/crm/class-crm-task-service.php
- includes/crm/class-crm-task-repository.php
- docs/contracts/43D.md
- docs/contracts/validation/43D-validation.md

==================================================
FORBIDDEN FILES
==================================================

- includes/modules/*
- schema/database
- includes/processes/*
- includes/appointments/*
- invoices/payments core

==================================================
OBJETIVO FUNCIONAL
==================================================

Agregar soporte mínimo de seguridad + rollback sobre acciones soportadas:

1. bulk_resolve
   - debe poder revertirse a `pending`
   - solo sobre tasks resueltas en la misma ejecución controlada

2. bulk_reassign
   - debe poder revertirse al `assigned_user_id` anterior
   - solo sobre tasks reasignadas en la misma ejecución controlada

3. reglas no soportadas
   - deben marcarse como no rollback-capable

==================================================
CRITERIO DE IMPLEMENTACIÓN
==================================================

- no duplicar lógica de mutación
- reutilizar servicios existentes
- registrar en memoria de ejecución / payload visible suficiente para revertir
- no crear tabla nueva de logs
- rollback solo si la acción fue ejecutada desde capa controlada
