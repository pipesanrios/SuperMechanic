# TASK CONTRACT - 57G-B Queue Retry Handling

## Objective
Implement controlled retry handling for the manual SaaS queue worker without enabling cron, scheduled workers, external calls or automatic execution.

## Scope
- Manual retry handling for persisted SaaS queue jobs.
- Repository/status behavior needed to preserve retry scheduling.
- Runtime smoke validation for retry transitions.
- Documentation updates.

- Current state, QA, roadmap, async queue and SaaS foundation docs.

## Constraints
- SQL is allowed only in `Queue_Job_Repository`.
- Worker must not register hooks, cron or schedules.
- Worker must not execute automatically.
- Worker processes at most one job per `process_next()` call.
- Retry handling is manual-only and uses persisted queue state.
- Failed processing increments attempts only on failure.
- Retry scheduling must clear lock state and store last error.
- Future `retry_scheduled` jobs must not be picked until `available_at <= now`.
- No real catalog writes, external calls, email sending, PDF generation or Google Calendar calls.

## Deliverables
- Retry behavior in `Queue_Worker`.
- Repository support for retry scheduling and future-available filtering.
- Deterministic retry backoff:
  - attempt 1: +5 minutes
  - attempt 2: +15 minutes
  - attempt 3+: +30 minutes
- Runtime smoke evidence.
- Documentation updated.

## Acceptance Criteria
- Failing job with attempts `0` transitions `running -> retry_scheduled` with attempts `1`.
- Future `retry_scheduled` job is ignored by `process_next()`.
- Due `retry_scheduled` job is picked by `process_next()`.
- Job reaching max attempts transitions `running -> failed`.
- `lock_token` is cleared after retry scheduling and final failure.
- No cron, scheduled event, hook, external API, real provider execution, catalog import, email sending, PDF generation, API endpoint or frontend asset is introduced.
