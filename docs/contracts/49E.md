PHASE: 49E
NAME: Consistency Hardening

==================================================
OBJETIVO
==================================================

Endurecer la consistencia del modelo de acceso multi-business
detectando y corrigiendo estados inválidos o ambiguos
entre roles globales, roles legacy y memberships por negocio.

==================================================
SCOPE
==================================================
- validación de consistencia de memberships
- validación de unicidad de primaria
- detección de usuarios con acceso indebido
- corrección segura de inconsistencias detectables
- warnings/estado más precisos en Roles & Access

==================================================
OUT OF SCOPE
==================================================
- migración masiva completa legacy -> memberships
- auditoría avanzada histórica
- permisos granulares por acción
- CRM Pipeline
- automatización compleja de corrección

==================================================
ALLOWED FILES
==================================================
- includes/users/class-business-membership-service.php
- includes/users/class-role-access-service.php
- includes/users/class-admin-roles-controller.php
- docs/contracts/49E.md
- docs/contracts/validation/49E-validation.md

==================================================
CRITERIO DE IMPLEMENTACIÓN
==================================================
- no romper datos existentes
- no borrar memberships automáticamente salvo corrección segura
- preferir warning + corrección controlada
- mantener compatibilidad con fase legacy actual
- centralizar validación de consistencia

==================================================
CASOS A DETECTAR
==================================================
- múltiples memberships primarias activas
- primaria inactiva
- membership activa duplicada para mismo user + business
- usuario con acceso interno sin membership activa
- mechanic sin business operativo
- client con acceso interno indebido
- usuario no global con scope multi-business inválido

==================================================
SERVICE REQUERIDO
==================================================
Agregar métodos o capa equivalente para:

- validate_membership_consistency($user_id)
- get_membership_consistency_warnings($user_id)
- repair_membership_consistency($user_id) // solo safe repairs

==================================================
SAFE REPAIRS PERMITIDAS
==================================================
- dejar una sola primaria
- desmarcar primarias inválidas
- forzar primaria activa si existe una válida
- limpiar duplicidad simple si es estrictamente segura
- NO inventar memberships nuevas automáticamente

==================================================
ACCEPTANCE CRITERIA
==================================================
- inconsistencias detectadas con claridad
- safe repairs funcionan
- warnings mejoran precisión
- sin regresión dashboard/roles
- modelo multi-business más robusto

==================================================
VALIDATIONS REQUIRED
==================================================

- QA runner
- runtime manual
