# TASK CONTRACT - 57F Queue Persistence Foundation

## Objective
Create persistent queue storage foundation for future SaaS jobs without activating workers, cron, external calls or real job execution.

## Scope
- SaaS queue persistence schema foundation.
- Queue job repository for create/read/list/status/retry operations.
- Optional persistence path in the passive queue dispatcher.
- Queue context persistence flag, disabled by default.
- Runtime/static validation that persistence works only when explicitly enabled.
- Documentation updates.

## In Scope
- `sm_saas_queue_jobs` schema/table registration.
- `Queue_Job_Repository` under `includes/saas/*`.
- `Queue_Context` persistence flag.
- `Queue_Dispatcher` optional persistence integration.
- Persistent queue job smoke validation.
- Current state, QA, roadmap, async queue and SaaS foundation docs.

## Out of Scope
- workers
- cron/scheduled execution
- real job execution
- external queue services
- external HTTP calls
- connector execution
- email sending
- API/admin endpoints
- frontend/assets
- CRM/users/process/payment logic

## Allowed Files
- includes/saas/*
- includes/database/*
- docs/*

## Forbidden Files
- CRM
- users/roles
- process/payment logic
- API controllers
- frontend/assets
- external integrations
- real worker execution
- cron activation

## Constraints
- SQL is allowed only in `includes/saas/class-queue-job-repository.php` and `includes/database/*`.
- Persistence must remain opt-in.
- Queue dispatcher default behavior must remain passive and non-persistent.
- No worker, cron, external calls or job execution may be introduced.
- `business_id` remains required for persisted tenant-aware queue jobs.
- Existing active queue/webhook behavior must not be migrated or replaced.

## Deliverables
- `sm_saas_queue_jobs` schema registered in the database layer.
- `includes/saas/class-queue-job-repository.php`
- Repository methods:
  - `create_job(array $job)`
  - `get_job_by_id($job_id)`
  - `list_jobs(array $filters = array())`
  - `update_status($job_id, $status, array $meta = array())`
  - `mark_failed($job_id, $error)`
  - `mark_completed($job_id)`
  - `schedule_retry($job_id, $available_at, $error = '')`
- Queue context persistence controls.
- Queue dispatcher optional persistence integration.
- Runtime smoke evidence.
- Documentation updated.

## Acceptance Criteria
- `sm_saas_queue_jobs` is registered with required columns and indexes.
- Repository can create, retrieve, list, update status, mark completed, mark failed and schedule retry for jobs.
- Dispatcher default remains passive and does not persist unless persistence is explicitly enabled.
- Dispatcher with persistence enabled can persist a normalized job without executing it.
- No cron, worker, external call, connector execution, email sending, API endpoint, frontend asset or forbidden module change is introduced.

  - no external calls
