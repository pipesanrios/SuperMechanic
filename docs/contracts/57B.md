# TASK CONTRACT — 57B Tenant Context Layer

## Objective
Introduce a passive tenant context layer that bridges future SaaS tenant identity with the current business-scoped runtime without changing existing business logic.

## Scope
- tenant context resolver
- business scope bridge
- passive helpers
- no runtime takeover yet

## In Scope
- Tenant_Context improvements
- Runtime_Context integration
- business_id bridge
- tenant_id placeholder strategy
- helper methods for future SaaS runtime

## Out of Scope
- database tenant split
- schema changes
- replacing business_id
- API redesign
- billing/subscriptions
- UI changes
- SaaS routing/subdomains

## Allowed Files
- includes/saas/*
- includes/helpers/* only if strictly needed
- docs/CURRENT_STATE.md
- docs/QA_REPORT.md
- docs/PLUGIN_ROADMAP.md
- docs/tasks/*

## Forbidden Files
- CRM
- process/payment logic
- users/roles
- API controllers
- database/schema files
- frontend/assets

## Constraints
- passive layer only
- business_id remains canonical active scope
- tenant_id remains future abstraction
- no behavioral runtime changes
- no schema changes

## Deliverables
- tenant context resolver
- business bridge helpers
- docs updated

## Acceptance Criteria
- `Tenant_Context` exposes:
  - `get_tenant_id()`
  - `get_business_id()`
  - `has_tenant()`
  - `has_business_scope()`
  - `to_array()`
  - `from_business_id($business_id)`
  - `from_runtime_context(...)`
- `Runtime_Context` exposes:
  - `get_mode()`
  - `is_self_hosted()`
  - `is_saas_future()`
  - `is_local_development()`
  - `get_tenant_context()`
- `business_id` remains the canonical current runtime scope.
- `tenant_id` remains a passive future SaaS abstraction.
- no runtime hooks, schema changes, API changes, frontend changes or billing behavior are introduced.
- documentation is updated with the bridge behavior.
