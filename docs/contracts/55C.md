PHASE: 55C
NAME: Webhooks / Event Dispatch Formalization

==================================================
OBJETIVO
==================================================

Formalizar un sistema de eventos salientes y webhooks
para integraciones externas, con nombres de evento,
payload y dispatch consistentes.

==================================================
SCOPE
==================================================
- catálogo formal de eventos
- event dispatch central
- integración con webhooks existentes
- payload estandarizado
- soporte para eventos clave del sistema

==================================================
OUT OF SCOPE
==================================================
- event bus complejo
- colas distribuidas
- replay de eventos
- firma avanzada por proveedor
- webhooks inbound
- UI compleja de observabilidad

==================================================
ALLOWED FILES
==================================================
- includes/webhooks/class-webhook-service.php
- includes/automation/class-automation-engine-service.php
- includes/integrations/webhooks/ (nuevo si hace falta)
- includes/class-plugin.php si hace falta
- docs/contracts/55C.md
- docs/contracts/validation/55C-validation.md

==================================================
EVENTOS REQUERIDOS
==================================================

Implementar al menos:

- process.created
- process.updated
- quote.approved
- invoice.paid
- payment.created

Opcional si queda limpio:
- client.created
- vehicle.created
- membership.transferred

==================================================
PAYLOAD REQUERIDO
==================================================

Formato estándar:

{
  "event": "process.created",
  "timestamp": "2026-04-09T10:00:00Z",
  "business_id": 3,
  "entity_type": "process",
  "entity_id": 123,
  "data": {...}
}

==================================================
SERVICE / DISPATCH
==================================================

Formalizar algo equivalente a:
- dispatch_event($event_name, $payload = [])
- normalize_event_payload(...)
- dispatch_webhook_event(...)

==================================================
CRITERIO DE IMPLEMENTACIÓN
==================================================
- no duplicar dispatch caótico
- centralizar nombres de evento
- payload consistente
- no romper webhooks actuales
- mantener compatibilidad con retry/queue/logging

==================================================
INTEGRACIÓN REQUERIDA
==================================================

Disparar eventos desde puntos limpios del sistema:
- creación/actualización de procesos
- aprobación de quote
- pago creado / factura pagada

==================================================
ACCEPTANCE CRITERIA
==================================================
- eventos se despachan correctamente
- nombres de evento consistentes
- payload útil y estable
- webhooks existentes siguen funcionando
- sin regresión del sistema

VALIDATIONS REQUIRED

- runtime manual
